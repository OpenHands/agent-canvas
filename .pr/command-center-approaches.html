🎛️ Command Center

Design exploration for an aggregated conversation dashboard in agent-canvas — showing agent status, finish actions, and latest messages across all running conversations at a glance.

POST-MVP · DESIGN PHASE

⚡ The Problem

When running multiple agent conversations simultaneously, there is no single view that shows what each agent is doing. Users constantly click between conversations just to check status — "Is it finished? Is it done yet?"

🔄

Context Switching

Clicking between conversations to check agent status wastes time and breaks flow.

🔇

No Finish Notifications

When an agent finishes, there's no notification unless you're watching that conversation.

📋

No Activity Summary

The sidebar shows titles and status dots — but not what the agent said or concluded.

⏱️

Latent Awareness

Sidebar polls every 10s for status. No push-based awareness of critical events like errors or completions.

Key constraint: WebSocket connections are 1:1 with conversations. The sidebar polls REST every 10s for conversation metadata (title, status, timestamps), but never fetches event content (messages, finish actions) for conversations you're not viewing.

🔬 Approaches

Six approaches ranging from quick sidebar enhancements to full-stack aggregation APIs. Click to expand.

A

Dashboard Route ★ Recommended MVP

Medium Effort ▶

Add a new top-level /dashboard route accessible from the sidebar (alongside Conversations, Skills, Settings). The page shows a table or card grid of all conversations with their status, last event timestamp, and latest agent message / finish action summary.

New Route REST Polling React Query Frontend Only

How It Works

  useDashboardData()  ← new React Query hook

  1. Poll searchConversations(limit=50) every 10s         (already exists)
  2. For each conversation with status ∈ {finished, error, running}:
     → Poll searchEvents(convId, {limit:1, sortOrder:TIMESTAMP_DESC})
     → Extract the latest FinishAction.message or last agent message
  3. Merge into a unified rows array:  { conversation, latestEvent, status }
  4. Render in a sortable/filterable table

  Optimization: Only fetch events for conversations updated since last poll
  (compare updated_at timestamps). Cache aggressively with React Query staleTime.

Key Implementation Details

// src/hooks/query/use-dashboard-conversations.ts
export function useDashboardConversations() {
  const conversations = usePaginatedConversations();

  // For each active/finished conversation, fetch latest event
  const eventQueries = useQueries({
    queries: (conversations.data?.pages.flatMap(p => p.items) ?? [])
      .filter(c => c.execution_status) // has a runtime status
      .map(conv => ({
        queryKey: ['dashboard', 'latest-event', conv.id, conv.updated_at],
        queryFn: () => EventService.searchEvents(
          conv.id, null, null,
          { limit: 3, sortOrder: 'TIMESTAMP_DESC' }
        ),
        staleTime: 15_000,
        enabled: !!conv.execution_status,
      })),
  });

  // Merge conversations with their latest events
  return useMemo(() => mergeConversationsWithEvents(
    conversations, eventQueries
  ), [conversations, eventQueries]);
}

Pros & Cons

  • Clear, dedicated space for multi-conversation oversight
  • No backend changes — uses existing APIs
  • Sortable/filterable table for power users
  • Natural extension of existing routing pattern
  • Can expand to show cost/token metrics per conversation
  • Works for both local and cloud backends
  • Not visible while inside a conversation (need to navigate)
  • N+1 queries: 1 for list + 1 per conversation for events
  • 10s polling latency for status changes
  • Could be slow with many conversations (mitigate with pagination)
B

Enhanced Sidebar Cards

Low Effort ▶

Extend the existing sidebar ConversationCard to show a preview of the last agent message — especially for FinishAction events. When a conversation finishes, the card expands slightly to show the finish message, highlighted with a green left border.

Sidebar Enhancement REST Polling Minimal New Code

How It Works

  Existing: ConversationCard shows title + status dot + timestamp

  Enhanced: ConversationCard + latest message preview
  ┌──────────────────────────────────┐
  │ Fix auth bug in user service     │
  │ ● Finished · 2 min ago              │
  │ ┌────────────────────────────── ┐│
  │ │ ✓ Fixed the JWT validation...  ││
  │ └────────────────────────────── ┘│
  └──────────────────────────────────┘

  Fetch: searchEvents(convId, {limit:1}) for conversations
  where execution_status ∈ {finished, error}
  Only fetch for recently-updated conversations (last 5 minutes)

Pros & Cons

  • Always visible — no need to navigate away
  • Minimal UI change, familiar pattern
  • Very low implementation effort
  • Could be a quick first step before Dashboard
  • Limited space — can only show truncated preview
  • Doesn't scale: 10+ conversations = lots of vertical scroll
  • Fetching events for all sidebar conversations is expensive
  • Mobile sidebar is even more constrained
C

Notification / Activity Feed Panel

Medium Effort ▶

Add a bell icon to the sidebar header that opens a slide-out activity feed. The feed aggregates important events across all conversations: FinishAction, errors, confirmation requests. Items link back to the originating conversation. An unread badge shows how many events happened since you last looked.

Overlay Panel Notification State Event Filtering Unread Tracking

How It Works

  ┌──────────────────────────────────────────────────────────────┐
  │  NotificationService (new singleton)                         │
  │                                                              │
  │  1. Subscribe to searchConversations() polling (already 10s)  │
  │  2. When a conversation's execution_status changes:          │
  │     → If new status = finished or error:                     │
  │       → Fetch searchEvents(convId, {limit:1})               │
  │       → Extract FinishAction.message or error details        │
  │       → Push to useNotificationStore (Zustand)               │
  │       → Increment unread badge count                         │
  │  3. When a conversation's status = waiting_for_confirmation:  │
  │     → Push a "needs your attention" notification             │
  │                                                              │
  │  UI: Bell icon → slide-out panel with chronological feed     │
  │  Clicking an item → navigate to that conversation            │
  └──────────────────────────────────────────────────────────────┘

Pros & Cons

  • Non-intrusive — works from any page
  • Unread badge provides at-a-glance awareness
  • Only fetches events on status transitions (efficient)
  • Familiar UX pattern (Slack, GitHub, etc.)
  • Could later integrate with OS notifications / browser Notification API
  • Requires new state management (notification store, read tracking)
  • Still depends on 10s polling to detect status changes
  • Feed can get noisy with many conversations
  • Need to handle persistence (which notifications were read?)
D

Multi-WebSocket Hub

High Effort ▶

Maintain WebSocket connections to all active conversations simultaneously, even when you're only viewing one. A background "hub" service opens WS connections, filters for important events (finish, error, confirmation), and aggregates them into a shared store.

Multi-WS Background Service Real-time Complex

How It Works

  WebSocketHub (new singleton service)

  ┌─── Browser ────────────────────────────────────────────────┐
  │                                                            │
  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐     │
  │  │ WS: Conv A   │  │ WS: Conv B   │  │ WS: Conv C   │ ... │
  │  │ (running)    │  │ (running)    │  │ (running)    │     │
  │  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘     │
  │         │                 │                 │              │
  │         └────────┬────────┴────────┬────────┘              │
  │                  ▼                 ▼                        │
  │        WebSocketHub.onEvent(convId, event)                 │
  │                  │                                         │
  │                  ▼                                         │
  │        useAggregatedEventStore                            │
  │        (cross-conversation event state)                    │
  │                                                            │
  │        ← powers Dashboard / Feed / Sidebar                 │
  └────────────────────────────────────────────────────────────┘

  Lifecycle:
  • Opens WS to each conv where execution_status ∈ {running, idle}
  • Closes WS when conversation finishes/errors (after capturing final events)
  • Reconnects on WS failure (reuse existing reconnect logic)
  • Filters events: only keeps FinishAction, AgentError, ConfirmationRequest

Pros & Cons

  • True real-time: instant finish/error awareness, no polling lag
  • Most responsive UX possible
  • Could power live "watching multiple agents code" view
  • Browser resource concern: 10+ simultaneous WS connections
  • Complex lifecycle management (open/close/reconnect per conv)
  • Backend may not expect many concurrent WS per session
  • Significantly more code to maintain and test
  • Mobile/low-bandwidth unfriendly
E

Polling-Based Event Aggregation

Low Effort ▶

A pragmatic middle ground: add a new React Query hook that piggybacks on the existing conversation list poll. For each conversation whose updated_at changed since last poll, fetch the last 1-3 events. Cache aggressively. No new backend APIs, no WebSockets — just smarter use of existing REST endpoints.

REST Only React Query Incremental No Backend Changes

How It Works

  useConversationEventPreviews()

  Dependencies:
    ← usePaginatedConversations()  (existing, 10s poll)

  Logic:
    1. On each conversations refetch, diff updated_at vs previous snapshot
    2. For each changed conversation:
       → searchEvents(convId, { limit: 1, sortOrder: TIMESTAMP_DESC })
       → Cache the result keyed by convId + updated_at
    3. Return Map<convId, { latestEvent, conversationType }>
    4. For unchanged conversations → return cached event (React Query staleTime)

  Consumer: any component (Sidebar card, Dashboard row, Feed item)

  Overhead:  worst case = N API calls (one per changed conversation)
            typical   = 1-3 calls per poll cycle (most conversations are idle)

Pros & Cons

  • Minimal new code — a single hook + cache strategy
  • Incremental: only fetches for conversations that changed
  • Powers all UI approaches (Dashboard, Sidebar, Feed)
  • No backend changes needed
  • Works with existing React Query infrastructure
  • Still 10s polling latency
  • N+1 request pattern during "burst" activity
  • Doesn't detect mid-poll events (e.g. finish between polls)
F

Server-Side Aggregation API

High Effort ▶

Add a new backend endpoint that returns the latest event per conversation in a single request. This eliminates the N+1 problem: one API call returns everything the command center needs. Could also add a "global events" WebSocket that multiplexes events from all conversations onto a single stream.

Backend Change New API Single Request Optimal Performance

Proposed API

// New endpoint on agent-server
GET /api/conversations/summary

// Response
{
  "items": [
    {
      "conversation_id": "abc-123",
      "title": "Fix auth bug",
      "execution_status": "finished",
      "updated_at": "2024-01-15T10:30:00Z",
      "latest_event": {
        "type": "FinishAction",
        "message": "Fixed JWT validation and added tests.",
        "timestamp": "2024-01-15T10:29:55Z"
      },
      "metrics": { "accumulated_cost": 0.42 }
    }
  ]
}

// Optional: global event stream WebSocket
WS /api/events/stream
// Multiplexes events from all running conversations
// with a conversation_id field on each event

Pros & Cons

  • Single request — no N+1, no polling overhead
  • Backend can optimize query (single DB scan)
  • Global WS multiplexer = true real-time without multi-WS
  • Cleanest long-term architecture
  • Requires agent-server changes (SDK contribution)
  • New API surface to maintain and version
  • Global WS multiplexer is complex to implement server-side
  • Coordination cost between frontend and backend teams

✅ Recommended Phased Approach

Rather than picking one approach, layer them progressively. Start with the data layer (E), add UI value quickly (B), then build the full dashboard (A) and notification system (C) as the feature matures.

Phase 1 · Quick Win

E + B: Data Hook + Sidebar Preview

Build useConversationEventPreviews() to incrementally fetch latest events. Wire it into the sidebar's ConversationCard to show finish/error message previews. ~1-2 days of work.

Phase 2 · Core Feature

A: Dashboard Route

Add /dashboard with a sortable table powered by the same data hook. Stats bar, filtering by status, click-to-navigate. ~2-3 days.

Phase 3 · Polish

C: Notification Feed + Browser Notifications

Add the bell icon + activity feed overlay. Unread tracking, conversation links, optional browser Notification API for finish events. ~2-3 days.

Phase 4 · Long-term

F: Server-Side Aggregation

Propose /api/conversations/summary endpoint to the SDK team. Replace the N+1 client-side polling with a single efficient API call. Optionally add a global events WebSocket.
